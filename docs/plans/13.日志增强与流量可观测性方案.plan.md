# ezlb 日志增强与流量可观测性方案

## 背景
- 当前配置中仅存在 `global.log_level` 与 `global.cleanup_on_exit`，尚未形成统一的 `global.log` 配置块。
- 现有日志能力以单一系统日志为主，缺少面向流量与 NAT 统计的独立日志输出能力。
- 需要以**极低开销**获取业务流量可见性，优先复用 IPVS / SNAT 现有统计信息，避免在转发热路径中引入逐请求埋点。
- 系统日志（System）、流量日志（Traffic）、NAT 日志（NAT）应具备清晰职责边界：
  - **System Logger**：用于启动、配置变更、reconcile、健康检查、异常等系统行为；
  - **Traffic Logger**：用于周期性输出 IPVS 流量增量统计；
  - **NAT Logger**：用于 SNAT 规则变更与 NAT 统计。
- 交付物包含：代码改动、本文档更新、README/README_CN 更新、示例配置更新、覆盖新能力的单测与 e2e / integration 测试。

## 目标
1. 建立统一的日志配置模型与 logger 构建方式，保证各模块日志字段一致、便于排障与采集。
2. 增加低开销流量日志能力，支持 **service 粒度的日志级别控制**。
3. 引入 NAT 统计日志，但在第一阶段明确按**规则/后端维度**输出；当多个 service 共享 backend 时，**不保证 service 级精确归因**。
4. 完成配置、文档、测试与运行行为定义，确保 macOS/Linux 开发环境与 Linux 运行环境均可验证。

## 非目标（Out of Scope）
本次方案**不包含**以下能力：

1. 请求级实时 access log；
2. 每连接级审计日志；
3. 在数据转发热路径中增加逐请求埋点；
4. 多 service 共享同一 backend 时的 NAT 精确 service 归因；
5. 完整替代 Prometheus / metrics 系统的长期监控体系。

## 配置升级说明

本次改造属于**日志配置模型升级**，不是在现有 `global.log_level` 上追加少量字段。

### 现状
当前 `pkg/config/config.go` 中仅包含：
- `global.log_level`
- `global.cleanup_on_exit`

### 目标结构
新增并统一使用（`traffic` 配置内聚到 `global.log` 下，保持日志配置单一入口）：

```yaml
global:
  log:
    level: info
    home: ./logs
    max_size: 50
    max_backups: 3
    max_age: 0
    compress: false
    traffic:
      enabled: true
      interval: 15s
      include_snat: true
```

> **设计决策**：`traffic` 本质是日志子系统的一部分，收纳到 `global.log` 下而非作为平级配置块，避免用户在两个不同层级查找日志相关配置。

### 迁移策略
- **不需要兼容旧字段**，直接使用新结构，在 README / 发布说明。
- `global.log.home` 如果未配置或者配置为空，使用默认值`./logs`。

## 总体实施策略

建议分两阶段实施，降低改造风险。

### Phase 1：日志基础设施改造
- 新增 `global.log` 配置块；
- 统一 logger 初始化；
- 引入 system / traffic / nat 三套 logger；
- 接入日志文件滚动；
- 更新 README / 示例配置 / 单测；
- 暂不引入完整 traffic collector。

### Phase 2：低开销流量与 NAT 可观测性
- 实现 IPVS 流量增量采集；
- 实现可选 SNAT 统计采集；
- 接入 traffic logger 周期输出；
- 接入 config 热更新；
- 补齐 Linux integration/e2e 测试。

#### 1.2 Logger 初始化（`cmd/ezlb/main.go` + 新增 `pkg/logutil`）

**职责划分**：
- `pkg/logutil` 负责提供 **logger 构建工具函数**（如 `NewSystemLogger`、`NewTrafficLogger`、`NewNATLogger`），封装 `zapcore.NewTee` + `lumberjack` 的组合逻辑。
- `cmd/ezlb/main.go`（或 `pkg/server`）负责 **logger 的生命周期管理**（创建、注入、Sync），保持现有的依赖注入风格。

**初始化时序（必须明确）**：
当前代码中 `config.NewManager()` 本身需要 logger 参数，因此 system logger 必须在配置加载之前就能创建。推荐的初始化流程：
1. `main.go` 中先用**默认配置**（info level + stdout）创建一个 bootstrap system logger；
2. 使用 bootstrap logger 初始化 `config.Manager` 并加载配置；
3. 读取 `global.log` 配置后，调用 `pkg/logutil` 构建正式的三套 logger（含文件输出、滚动等）；
4. 将正式 logger 注入到 server 及各模块；bootstrap logger 完成使命后 Sync 并丢弃。

**已有 bug 修复**：当前 `cmd/ezlb/main.go` 中的 `newLogger()` 硬编码 `zap.InfoLevel`，从未读取 `global.log_level`（现为 `global.log.level`）配置。本次改造需一并修复，确保 logger 级别与配置一致。

- 设计 `buildLoggers(cfg LogConfig)`，统一创建三套 logger：
  - **System Logger**
  - **Traffic Logger**
  - **NAT Logger**
- 使用 `zapcore.NewTee` + `lumberjack` 构建多输出 core。
- 文件名本期固定为：
  - `ezlb.log`
  - `traffic.log`
  - `nat.log`
- 输出规则：
   - System Logger → stdout/stderr + `${home}/ezlb.log`
   - Traffic Logger → `${home}/traffic.log`
   - NAT Logger → `${home}/nat.log`
- 文件打开 / 目录创建失败时的策略：
  - 优先记录告警；
  - 回退到 stdout/stderr；
  - **不建议**因 traffic/nat 日志文件创建失败直接阻断服务启动。

> **注意**：当前 CLI 中不存在 `--log-level` 参数（日志级别一直通过配置文件控制），无需删除。

### 2. 低开销流量日志

#### 2.1 配置扩展（`pkg/config/config.go`）
- `ServiceConfig.traffic_log_level` 默认继承 `global.log.level`。
- 当 `traffic_log_level == none` 时，该 service 不输出 traffic log。
- `global.log.traffic.enabled=false` 时，不启动 traffic collector。

#### 2.2 统计数据模型与接口约束

在编码前，需先定义 stats provider 接口与数据模型，避免实现阶段各模块各自推断。

建议引入：

```go
type ServiceTrafficStats struct {
    ServiceKey        string
    Connections       uint64
    InPkts            uint64
    OutPkts           uint64
    InBytes           uint64
    OutBytes          uint64
}

type BackendTrafficStats struct {
    ServiceKey        string
    BackendKey        string
    Connections       uint64
    InPkts            uint64
    OutPkts           uint64
    InBytes           uint64
    OutBytes          uint64
}

type SNATRuleStats struct {
    RuleKey           string
    BackendKey        string
    Packets           uint64
    Bytes             uint64
}

type TrafficSnapshot struct {
    Services map[string]ServiceTrafficStats
    Backends map[string]BackendTrafficStats
    SNAT     map[string]SNATRuleStats
}
```

接口建议：

```go
type LVSStatsProvider interface {
    ServiceStats() (map[string]ServiceTrafficStats, error)
    BackendStats() (map[string]BackendTrafficStats, error)
}

type SNATStatsProvider interface {
    Stats() (map[string]SNATRuleStats, error)
}
```

**LVSStatsProvider 实现策略**：
现有 `IPVSHandle.GetServices()` 和 `IPVSHandle.GetDestinations()` 已经返回了 `SvcStats` / `DstStats`（`ipvs_handle_linux.go` 中 `fromMobyService` / `fromMobyDestination` 已填充统计字段）。因此：
- **推荐**：基于 `lvs.Manager` 实现一个适配器（adapter），直接复用 `GetServices()` / `GetDestinations()` 获取统计数据，**不修改 `IPVSHandle` 接口**（避免 fake 实现同步改动）。
- 示例：
  ```go
  // pkg/trafficlog/lvs_stats_adapter.go
  type lvsStatsAdapter struct {
      manager *lvs.Manager
  }
  func (a *lvsStatsAdapter) ServiceStats() (map[string]ServiceTrafficStats, error) { ... }
  func (a *lvsStatsAdapter) BackendStats() (map[string]BackendTrafficStats, error) { ... }
  ```

**类型精度说明**：现有 `SvcStats.Connections` 为 `uint32`（来自 moby/ipvs），而本方案中 `ServiceTrafficStats.Connections` 定义为 `uint64`。实现时需注意 `uint32 → uint64` 的转换，`uint64` 范围更大，适合增量计算场景。

**Fake 环境行为（macOS 开发）**：
- `fakeHandle` 当前不产生真实统计数据（CRUD 操作不更新 Stats 字段），因此 fake 环境下 `LVSStatsProvider` 返回的统计值全为 0。
- Traffic collector 在 fake 环境下正常运行，但因 delta 始终为 0，不会输出任何 traffic 日志（符合"仅在有变化时写日志"的规则）。
- 这保证了 macOS 开发环境下 collector 逻辑可被测试（启动/停止/配置变更），但不会产生噪音日志。

同时定义以下行为：
- 首次采样仅建立基线，不输出 delta；
- 计数回绕 / 清零时视为基线重置，不输出负增量；
- rule 消失 / 重建时，重新建立基线；
- “仅在有变化时写日志”指 delta 任一关键值非 0（如 `connections_delta` / `bytes_in_delta` / `bytes_out_delta`）。

#### 2.3 统计收集组件（`pkg/trafficlog/`）

> **包路径决策**：使用 `pkg/trafficlog/` 而非 `pkg/telemetry/trafficlogger/`，与现有的 `pkg/snat/`、`pkg/lvs/` 等扁平结构保持一致。当前项目中没有其他 telemetry 子模块，无需提前引入 `pkg/telemetry/` 层级。
- 依赖 `LVSStatsProvider` 与可选 `SNATStatsProvider`。
- 维护快照并按间隔计算增量。
- 按 service 配置决定：
  - 是否输出；
  - 输出级别（debug/info/warn/error/none）。
- 默认只在有变化时输出。
- 若 stats 获取失败：
  - 记录 system logger warn/error；
  - 不中断主流程；
  - 下个周期继续采集。

#### 2.4 SNAT 统计边界（必须明确）

**接口设计决策**：`SNATStatsProvider` 应作为**独立接口**定义（如 `pkg/snat/stats.go`），**不修改现有 `snat.Manager` 接口**。原因：
- 当前 `snat.Manager` 是一个干净的声明式 reconcile 接口（`Reconcile` / `ReconcileForward` / `Cleanup`），加入 `Stats()` 会混入查询职责。
- Go 的隐式接口天然支持 `linuxManager` 同时实现 `Manager` 和 `StatsProvider`，而 `FakeManager` 不需要被迫实现 `Stats()`。
- Traffic collector 通过类型断言（`if sp, ok := snatMgr.(snat.StatsProvider); ok`）判断是否支持统计采集。

```go
// pkg/snat/stats.go — 独立接口，不修改 Manager
type StatsProvider interface {
    Stats() (map[string]SNATRuleStats, error)
}
```

- Linux 实现（`linuxManager`）解析 `iptables -t nat -vnL EZLB-SNAT`（或 nft 兼容路径）实现 `StatsProvider`。
- Fake 实现不实现 `StatsProvider`，collector 通过类型断言跳过 SNAT 统计采集。
- **第一阶段 NAT 统计按规则/后端维度输出，不承诺 service 粒度精确归因。**
- 原因：当前 SNAT rule key 仅由 `backendIP/backendPort/protocol` 组成；若多个 service 共享 backend，iptables 计数无法天然拆分到 service。
- 因此日志文案和 README 中需明确：
  - NAT 统计适合作为 backend / rule 层面的可观测性补充；
  - 不应被解读为跨共享 backend 场景下的精确 service 统计。

#### 2.5 Server 集成（`pkg/server/server.go`）

**运行模式区分**：
- **daemon 模式（`Run()`）**：启动 traffic collector goroutine。
- **once 模式（`RunOnce()`）**：执行一次 reconcile 后立即退出，**不启动 traffic collector**。

**注入依赖**：
- traffic logger
- nat logger
- stats provider（`LVSStatsProvider` + 可选 `SNATStatsProvider`）
- 当前配置快照

**配置热更新路径**：
当前 `config.Manager.OnChange()` 返回 `chan struct{}`，不携带变更内容。推荐采用**主动推送模式**（与现有 `healthMgr.UpdateTargets()` 模式一致）：
- Server 在 `OnChange` 回调中读取新配置快照，主动调用 collector 的更新方法（如 `collector.UpdateConfig(newCfg)`）。
- Collector 不应自行持有 config 引用或自行监听变更。
- 动态刷新的配置项：
  - `global.log.traffic.enabled`
  - `global.log.traffic.interval`
  - `global.log.traffic.include_snat`
  - `service.traffic_log_level`

**Shutdown 行为**：
- `shutdown()` 时停止 collector goroutine。
- 三套 logger 的 `Sync()` 必须在 `shutdown()` 中调用，确保日志刷盘后再退出。建议在 `server.shutdown()` 末尾统一调用，或由 `main.go` 通过 `defer logger.Sync()` 保证。

#### 2.6 日志输出建议字段

建议 traffic / nat 日志至少包含：
- `service`
- `listen`
- `protocol`
- `scheduler`
- `backend`（如适用）
- `interval`
- `source`（`ipvs` / `snat`）
- `connections_delta`
- `bytes_in_delta`
- `bytes_out_delta`
- `packets_in_delta`
- `packets_out_delta`

NAT 日志建议至少包含：
- `backend`
- `rule_key`
- `snat_ip`
- `packets_delta`
- `bytes_delta`
- `source=snat`

## 最终结论

该方案可以推进，但必须以"**日志基础设施先行、流量统计增强随后落地**"为原则实施。

编码前应以本修订版为准，特别注意以下约束：
1. `global.log` 是配置模型升级，`traffic` 配置内聚到 `global.log.traffic` 下；
2. `global.log.home` 默认值为 `./logs`；
3. Logger 初始化需处理时序问题：bootstrap logger → 加载配置 → 正式 logger；
4. 修复现有 `log_level` 未生效的 bug（`main.go` 中硬编码 InfoLevel）；
5. `LVSStatsProvider` 基于 `lvs.Manager` 适配器实现，不修改 `IPVSHandle` 接口；
6. `SNATStatsProvider` 作为独立接口，不修改 `snat.Manager` 接口；
7. NAT 统计第一阶段按规则/后端维度输出，不承诺 service 粒度精确归因；
8. Traffic collector 仅在 daemon 模式启动，once 模式不启动；
9. 新增 `gopkg.in/natefinch/lumberjack.v2` 依赖；
10. 必须先定义 stats provider 接口与快照模型，再进入实现。
