- 实现提交：

## 总体结论
本轮改动已经把统一日志配置、三类 logger、traffic collector 与 SNAT stats 的主体框架搭起来了，但当前实现里仍有几处会直接影响功能可用性的缺陷，暂时不建议判定为“实现正确”。

## 主要问题

### 1. `global.log.traffic.enabled` 关闭启动后，热更新为开启不会真正启动 collector
- 位置：`pkg/server/server.go`
- 现状：`Run()` 只在启动阶段根据初始配置创建 `s.collector`；后续热更新阶段仅在 `s.collector != nil` 时调用 `UpdateConfig()`。
- 影响：如果进程启动时 `global.log.traffic.enabled: false`，后续把配置改成 `true` 也不会创建 collector，traffic / nat 日志会一直缺失，必须重启进程才能生效。

### 2. SNAT manager 仍绑定 system logger，`nat.log` 会漏掉大部分 NAT 操作日志
- 位置：`pkg/server/server.go`
- 现状：`snat.NewManager()` 仍然传入 `logger.Named("snat")`，没有使用专门的 NAT logger。
- 影响：FullNAT 下建链、删链、规则增删、cleanup 等日志仍写入 `ezlb.log` / stdout，而 `nat.log` 只剩 collector 的统计输出，不符合方案中 “SNAT 规则变更与 NAT 统计使用 NAT logger” 的设计目标。

### 3. Linux 上 SNAT 统计解析逻辑会把所有规则都跳过
- 位置：`pkg/snat/manager_linux.go`
- 现状：当前代码按 `stat[9:]` 逐元素查找独立的 `dpt:` token；但 `go-iptables` 的 `Stats()` 已经把额外选项合并到了 `stat[9]` 一个字符串中。
- 影响：`dport` 基本无法解析出来，所有规则都会被 `continue` 掉；开启 `include_snat` 后，`nat.log` 仍然拿不到任何 SNAT 统计数据。

### 4. 日志目录已存在但不可写时，traffic/nat logger 不会按预期回退到 stdout
- 位置：`pkg/logutil/logutil.go`
- 现状：当前只在 `os.MkdirAll(home)` 失败时才走 fallback；如果目录已存在但没有写权限，会进入 file-only 分支，直到首条日志写入时才由 lumberjack 报错。
- 影响：这种场景下 `traffic.log` / `nat.log` 的日志会直接丢失，和代码注释里“失败后回退到 stdout”的行为不一致，在只读挂载目录场景下尤其明显。

## 建议
1. 在 `OnChange` 路径里补齐 collector 的创建、停止与重建逻辑，而不是只更新已有实例。
2. 将 `snat.Manager` 的业务日志统一切到 NAT logger，保持 `ezlb.log` / `nat.log` 职责边界清晰。
3. SNAT 统计解析优先改用 `StructuredStats()`，或至少按 `stat[9]` 的完整 option 字符串解析 `dpt:`。
4. 为“目录存在但不可写”的场景补单测，并把 fallback 逻辑放到真正能探测文件可写性的路径上。
